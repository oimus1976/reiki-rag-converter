## 0. 本文書の位置づけとスコープ

本書は、  
Phase 1（条例本文の構造保持を目的とした Markdown 化）および  
Phase 2（別表・数表を対象とした CSV 抽出）の成果を、  
**実運用における回答品質評価へ正しく接続するための  
評価設計ブリーフ**である。

本フェーズ（Phase 3）では、  
Markdown 単体利用と、Markdown＋CSV 併用時の回答を比較し、  
**どちらが「より正しい回答」を生みやすいかを  
評価可能な形で言語定義する**ことを目的とする。

ただし、本書は以下を行わない。

- 正解データセットの作成
- 正答率・スコアリングによる評価
- RAG 構成・チャンク設計・検索チューニング
- CSV / Markdown 仕様の改修や最適化

本書の役割は、  
**評価を実施するための前提・評価軸・観測観点を固定すること**に限定される。

本書で定義される評価設計は、  
Phase 3.5 / Phase 4 における実験・実装フェーズにおいて  
再利用・検証される前提資料として扱われる。

## 1. Phase 3 の目的と完了条件

Phase 3 の目的は、  
**CSV を併用することで、実運用における回答が  
「より正しいものになり得るか」を評価可能な形で定義すること**にある。

ここでいう「より正しい回答」とは、  
単純な正答率や点数による優劣ではなく、  
**誤回答が発生した場合の性質がどのように変化するか**  
— とりわけ、実務上問題となる**致命的な誤り** —  
に着目した概念である。

Phase 3 では、  
Markdown 単体利用と Markdown＋CSV 併用時の回答を比較し、  
評価シナリオに基づいて、

- どのような誤回答が生じ得るか
- その誤りがどの評価軸において、どのように変化するか

を**言語で記述可能な状態にする**ことをゴールとする。

本フェーズの完了条件は、  
評価シナリオを通じて、  
CSV 併用の有無による回答差分を  
**評価軸に沿って説明できていること**であり、  
優劣判定・合否判定・最適化判断を行うことは含まれない。

## 2. 前提条件と責務境界

本書は、Phase 1（Markdown）および Phase 2（CSV）で確定した  
**既存の設計・成果物を前提条件として受け入れる**。  
本フェーズにおいて、それらの仕様・構造・スキーマを再検討・改変することは行わない。

また、本書は既存の **Evaluation Framework** の評価思想および  
責務分離（Observation / Evaluation / Judgment）を尊重し、  
これを変更・上書きするものではない。

Phase 3 における責務は、  
**評価を実施するための前提条件と評価観点を言語で固定すること**に限定される。  
そのため、以下の事項は本フェーズのスコープ外とする。

- 正解データセット（Golden Answer 等）の作成
- 正答率・点数・スコアリングによる評価
- RAG の構成変更、チャンク設計、検索条件の最適化
- CSV / Markdown の構造・内容・出力仕様の修正
- モデル選択やモデル間性能比較

本フェーズで行う評価設計は、  
**「評価できる状態を作る」ことを目的とした準備工程**であり、  
評価結果の良否判断や設計上の是非を確定するものではない。

Phase 3 の成果は、  
後続フェーズ（Phase 3.5 / Phase 4）において  
実験・実装・判断を行うための**前提資料**としてのみ利用される。

## 3. 「より正しい回答」の定義（Phase 3）

本フェーズ（Phase 3）において扱う「より正しい回答」とは、  
**正答率や点数によって測定される正しさ**を指すものではない。

Phase 3 では、回答の正しさを  
**誤回答が発生した場合に、その性質がどのようなものになるか**  
という観点から捉える。

特に重視するのは、以下のような  
**実運用上、影響が大きい誤回答が回避・抑制されるかどうか**である。

- 金額・数値・区分などの取り違え
- 条文上の条件（適用範囲・前提条件）に対する誤認
- 条文と別表（表・数表）との対応関係の誤り

これらの誤りは、  
回答が一見もっともらしく見える場合であっても、  
実務上は **致命的な誤回答** となり得る。

本フェーズでは、  
Markdown 単体利用と Markdown＋CSV 併用時の回答を比較し、  
**致命的な誤回答がどのように生じ、  
併用によってその発生様式がどのように変化するか**  
を観測・記述可能な形で定義する。

したがって、Phase 3 における「より正しい回答」とは、  
「常に正解する回答」や  
「誤りが一切含まれない回答」を意味するものではなく、  
**誤りが生じた場合でも、  
実運用上のリスクが相対的に低い状態に近づいている回答**  
を指す概念として用いられる。

## 4. 評価軸の構成

Phase 3 における評価は、  
単一の指標や点数によって行うものではなく、  
**複数の評価軸に基づいて回答の性質を観測・記述する**方式を採用する。

評価軸は、  
**判定に直接用いる主評価軸**と、  
**判断の補助材料として扱う観測補助軸**に分けて定義する。

---

### 4.1 主評価軸（判定に用いる）

主評価軸は、  
Markdown 単体利用と Markdown＋CSV 併用時の回答差分を比較する際に、  
**評価シナリオごとの変化を整理するための中核的観点**である。

#### 致命誤り回避性

金額・数値・区分・条件などに関して、  
実運用上重大な影響を及ぼす誤回答が  
どの程度回避・抑制されているかを評価する。

#### 条件対応づけ精度

条文中に記載された条件と、  
別表・数表における行・列・区分との対応関係が、  
正しく結び付けられているかを評価する。

#### 参照行動の健全性

質問内容に応じて、  
条文・別表のいずれを参照すべきかという判断が適切に行われ、  
片方の情報だけに引きずられた回答になっていないかを評価する。

---

### 4.2 観測補助軸（判断材料）

観測補助軸は、  
主評価軸の判定を補強するための材料として位置づける。  
単独での良否判定や優劣評価には用いない。

#### 説明可能性（条文＋表の併用状況）

回答において、  
条文の記述と別表（CSV）の内容がどのように併用され、  
「なぜその回答になるのか」を説明できているかを観測する。

説明可能性は、  
回答の分かりやすさや文章表現の巧拙を評価するものではなく、  
**参照関係が可視化されているかどうか**を確認するための補助的観点である。

---

## 5. 評価対象の比較構成

Phase 3 における評価は、  
回答生成時に用いる入力構成の違いによって生じる差分を把握するため、  
**以下の2構成を比較対象**とする。

- **A：Markdown 単独**  
  条文構造を保持した Markdown のみを入力として用いる構成。

- **B：Markdown ＋ CSV 併用**  
  条文構造を保持した Markdown に加え、  
  別表・数表を CSV として併用する構成。

比較の対象は、  
**入力として与えられる情報構造の違い**に限定される。  
そのため、本フェーズでは以下を行わない。

- CSV スキーマの変更・拡張
- CSV の分割単位・チャンク設計の変更
- RAG 構成、検索方式、プロンプト設計の変更
- モデルの切替やモデル間比較

あくまで、  
**同一の評価条件下において、  
Markdown 単独利用と CSV 併用が  
回答の性質にどのような差をもたらすか**を観測・記述することを目的とする。

---

## 6. 評価シナリオ設計方針

Phase 3 における評価シナリオは、  
Markdown 単体利用と Markdown＋CSV 併用時の差分を  
適切に観測できるよう設計する。

評価シナリオの選定にあたっては、  
**Markdown 単体でも一見すると妥当に見えるが、  
実運用においては誤回答が発生し得るケース**を  
優先的に対象とする。

これは、  
Markdown 単体利用が常に不十分であることを示すためではなく、  
**一見問題がなさそうな回答が、  
どのような条件下で誤りに転じるか**を明らかにするためである。

そのため、評価シナリオは  
「CSV 併用が有効であることを示す例」や  
「CSV 併用時にのみ正解できる例」を集めることを目的としない。

評価シナリオは、  
回答生成時に以下のような判断を要するケースを想定する。

- 条文上の条件と、別表・数表の行・列との対応関係が問われる場合
- 複数の金額・区分が存在し、代表値による誤認が起きやすい場合
- 条文のみ、または表のみを参照すると誤りに至る場合

これにより、  
Markdown 単体利用と CSV 併用の差分を、  
**評価軸に基づいて比較・記述可能な状態**を作ることを目指す。

---

## 7. 最小評価シナリオ一覧

本章では、  
Phase 3 において用いる **最小限の評価シナリオ** を定義する。  
各シナリオは、Markdown 単体利用と Markdown＋CSV 併用時の差分を  
評価軸に基づいて観測・記述するための共通土台として位置づける。

評価シナリオは、  
網羅性や代表性を目的とするものではなく、  
**誤回答の性質変化を捉えやすい典型例**として選定する。

---

### 7.1 シナリオ S1（条件 × 金額表）

条文中に記載された条件と、  
別表に示された複数の金額・数値との対応関係が  
正しく解釈されることが求められるケース。

本シナリオでは、  
条文条件を十分に解釈しないまま表を参照した場合や、  
表中の金額を代表値として扱った場合に、  
誤回答が生じ得る状況を想定する。

Markdown 単体利用と CSV 併用時で、  
条件と金額の対応づけが  
どの評価軸において、どのように変化するかを観測する。

---

### 7.2 シナリオ S2（区分・代表値誤認）

別表に複数の区分・段階・条件付き金額が存在し、  
Markdown 単体利用では  
特定の区分や代表的な値に引きずられやすいケース。

本シナリオでは、  
条文上の適用区分を十分に考慮せずに回答した場合に、  
一見もっともらしいが不適切な回答が生成される状況を想定する。

Markdown 単体利用と CSV 併用時で、  
区分認識および参照行動が  
どのように変化するかを観測する。

---

### 7.3 シナリオ S3（implicit annex ＋ 表）

別表や付表が、  
明示的な参照関係を持たず、  
暗黙的に条文に帰属しているケース。

本シナリオでは、  
附属表の位置づけや適用範囲を誤認した場合に、  
誤った前提で回答が生成される状況を想定する。

本シナリオは任意とし、  
必要に応じて評価対象に含めることで、  
implicit annex 構造と表参照の組み合わせに対する  
耐性を観測する。

---

## 8. 評価の進め方（判定しないルール）

Phase 3 における評価は、  
Markdown 単体利用と Markdown＋CSV 併用時の回答を比較し、  
**評価軸ごとの変化を記述すること**を目的とする。

本フェーズでは、以下を行わない。

- 回答の優劣判定
- 合否判定（OK / NG 等）
- 正解・不正解の確定
- 点数化・スコアリング

評価は、  
各評価シナリオについて、  
主評価軸および観測補助軸に基づき、

- どのような誤回答が生じ得るか
- その誤りがどの評価軸に影響しているか
- Markdown 単体利用と CSV 併用で、  
  参照行動や誤りの性質がどのように変化するか

を **文章で整理・記録する** 形で行う。

この記述は、  
後続フェーズにおける実験・実装・判断のための  
**観測結果の整理**を目的としたものであり、  
CSV 併用の是非や最適解を  
この時点で確定するものではない。

Phase 3 の評価記録は、  
**「判断のための材料」**としてのみ扱われ、  
Evaluation Framework における Gate 判定や  
Judgment 文書に直接接続されるものではない。

なお、本フェーズにおける記述は、  
評価軸および評価シナリオに沿った  
**構造化された記述**を前提とし、  
自由形式の所感記述を目的とするものではない。

Phase 3 は、  
既存の **Evaluation Framework** における  
Gate 判定の **前段階** に位置づけられる評価設計フェーズである。

本フェーズの目的は、  
Gate 判定を行うための材料を整備することであり、  
Gate 判定そのものや、  
評価結果の是非を確定することではない。

そのため、Phase 3 で定義される評価軸・評価シナリオ・記述方針は、  
Framework v0.x における評価思想および  
責務境界（Observation / Evaluation / Judgment）を  
**変更・上書きするものではない**。

Phase 3 の成果は、  
後続フェーズにおいて、

- 実験設計の基礎資料
- 評価観測項目の整理

として利用されることを想定するが、  
Phase 3 単体で  
Gate 判定に直接接続されることはない。

この位置づけにより、  
Phase 3 は  
**評価を実行する前に、
評価の考え方を固定するための準備工程**  
として機能する。
